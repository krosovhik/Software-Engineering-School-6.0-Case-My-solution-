# GitHub Releases Subscription Service

Монолітний сервіс для автоматичного відстеження нових релізів у GitHub репозиторіях та надсилання email сповіщень підписникам

---

## Технологічний стек
* **Backend:** Node.js (тонкий фреймворк Fastify/Express)
* **Database:** PostgreSQL (з автоматичними міграціями)
* **Caching:** Redis (кешування GitHub API, TTL 10 хв)
* **Testing:** Jest (Unit-тести на бізнес-логіку)
* **Monitoring:** Prometheus (ендпоїнт `/metrics`)
* **Infrastructure:** Docker & Docker Compose
* **Mail Testing:** MailHog (локальний SMTP сервер)

---

## Архітектура та Модулі
Згідно з вимогами, сервіс реалізовано як **моноліт**, що складається з трьох логічних частин:

1. **API Layer**: Обробка HTTP-запитів, керування підписками та надання Swagger документації.
2. **Scanner**: Фоновий процес, що періодично опитує GitHub API для виявлення нових тегів.
3. **Notifier**: Модуль формування та відправки email-сповіщень.

---

## Основна логіка роботи

### Підписка та Валідація
* При створенні підписки сервіс перевіряє формат `owner/repo` та існування репозиторію через GitHub API.
* **Smart Start**: Поточний реліз репозиторію зберігається як `last_seen_tag`, але сповіщення не надсилається відразу. Користувач отримає лист лише тоді, коли з'явиться **наступний**,тобто новий реліз.

### Сканування та Оптимізація
* Scanner групує всі активні підписки за репозиторіями, щоб мінімізувати кількість запитів до GitHub.
* Використовується **Redis-кешування** для відповідей GitHub API, що критично для обробки лімітів (Rate Limit).
* Реалізовано коректну обробку помилки `429 Too Many Requests`.

## Використання (API)

Документація Swagger доступна за адресою: `http://localhost:3000/docs`

| Метод  | Ендпоїнт | Опис |
| :--- | :--- | :--- |
| **POST** | `/subscriptions` | Створити нову підписку |
| **DELETE** | `/subscriptions` | Видалити підписку |
| **GET** | `/subscriptions?email=...` | Список підписок за email |
| **GET** | `/health` | Перевірка стану сервісу |
| **GET** | `/metrics` | Метрики для Prometheus |
| **POST** | `/scanner/run` | Ручний запуск сканування (force run) |

## Запуск проєкту

### Повний запуск у Docker (Рекомендовано)
Найшвидший спосіб підняти всю інфраструктуру (App, DB, Redis, MailHog):
```bash
cp .env.example .env
docker compose up --build
```
### Локальна розробка
Якщо потрібно запустити сервіс локально:
1. **Встановити залежності:** `npm install`
2. **Підняти тільки БД та кеш:** `docker compose up -d db redis mailhog`
3. **Запустити сервіс:** `npm start`
