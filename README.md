# GitHub Releases Subscription Service

Монолітний сервіс, що дозволяє підписуватись на email-сповіщення про нові релізи GitHub-репозиторіїв.

---

## Технології

- Node.js (Fastify / Express)
- PostgreSQL
- Redis
- Docker / Docker Compose
- Jest (unit-тести)
- Prometheus (/metrics)
- MailHog (email тестування)

---

## Що реалізовано

- REST API + Swagger UI (`/docs`)
- HTML-сторінка (`/`) для тестування
- Валідація репозиторію (`owner/repo`)
- Зберігання даних у PostgreSQL
- Міграції БД при старті
- Scanner (фоновий процес)
- Notifier (email)
- Обробка `429 Too Many Requests`
- `last_seen_tag` для кожного репозиторію
- Unit-тести

---

## Extra features

- Redis-кешування (TTL 10 хв)
- API key auth (`x-api-key`)
- Prometheus-метрики
- GitHub Actions CI

---

## Архітектура

Сервіс реалізовано як моноліт (згідно вимог).

Складається з 3 логічних частин:

### API Layer
- обробка HTTP-запитів
- керування підписками
- Swagger

### Scanner
- періодично перевіряє GitHub API
- визначає нові релізи

### Notifier
- надсилає email користувачам

---

## Основна логіка

### Створення підписки

1. Перевірка формату `owner/repo`
2. Перевірка існування через GitHub API
3. Створення або активація підписки

---

### Робота scanner

1. Отримання всіх активних підписок
2. Групування по репозиторіях
3. Отримання latest release
4. Порівняння з `last_seen_tag`

Якщо реліз новий:
- надсилається email
- оновлюється `last_seen_tag`

---

### Важлива поведінка

При створенні підписки:
- поточний реліз зберігається
- email не надсилається

👉 Це запобігає дублюванню старих релізів  
👉 Користувач отримує тільки нові релізи після підписки

---

## Кешування

- GitHub API кешується в Redis (10 хв)
- зменшує кількість запитів
- знижує ризик rate limit
